# Spec 008 — الصندوق وسندات القبض، صلاحيات الذكاء الاصطناعي، وسجل نشاط الموظفين

**الحالة:** مسودة للمراجعة (قبل أي كود)
**المصدر:** ملاحظات العميل عبر واتساب (2026-07-20) + قرارات نطاق من المالك
**الهدف على السيرفر:** التطبيق ثم النشر على MyContabo (91.230.110.187:9095)

---

## 0. الخلفية والمشكلة

نظام دفعات الإيجار الحالي (`shop_rentpay`) فيه زر واحد يقلب حالة الدفعة بين `مدفوع`/`غير مدفوع`
عبر `POST /shop/toggle_rentpay` → `ShopController::toggle_rentpay()` (`ShopController.php:735`).
الأثر الوحيد هو تحديث `update_user` + `paid_date` على نفس الصف — **لا سند قبض، لا سجل غير قابل للحذف،
لا صندوق يتابع**. أي موظف صاحب صلاحية (function 33) يقلبها بصمت، وهذا يفتح باب التلاعب الذي حذّر منه العميل:

> «بس المفروض جنب كل دفعة سند قبض، مو فقط تحويلها إلى مدفوع وغير مدفوع، لأنه ممكن يصير في تلاعب
> من أي موظف يقوم بتغييرها بدون علم الثاني. المفروض بس تتغير حالتها يكون هناك مثل الصندوق يتابع
> حالة المدفوع وغير مدفوع.»

يتفرّع الطلب إلى **ثلاث مهام** مستقلة نسبياً.

## المهمة 1 — الصندوق الموحّد + سندات القبض (السند = مو مجرد زر)

### القرارات المعتمدة
- **النطاق:** صندوق **موحّد لكل الحركات المالية** — الإيجار (`shop_rentpay`) + المصاريف (`expense`)
  + المشتريات (`purchase`) + `accountings` + `financial` + النظام الموازي `lease_payments`.
- **عند الإلغاء (رجوع لغير مدفوع):** سجل **غير قابل للحذف** + **سبب إلزامي**. لا حذف صامت — قيد عكسي فقط.
- **الطباعة:** كل سند قبض له صفحة **PDF قابلة للطباعة** برقم متسلسل (DOMPDF موجود بالمشروع).

### المتطلبات
1. **جدول سندات القبض `cash_receipt`** (سند قبض دائم):
   - `receipt_id` PK
   - `receipt_no` — رقم متسلسل فريد قابل للطباعة (لا يُعاد استخدامه)
   - `source_type` — نوع الحركة (`shop_rentpay` | `expense` | `purchase` | `accountings` | `financial` | `lease_payment`)
   - `source_id` — مفتاح الصف المصدر
   - `direction` — `in` (قبض) | `out` (صرف)
   - `amount` — المبلغ الفعلي المقبوض/المصروف (مستقل عن المبلغ المجدول)
   - `receipt_date`, `payer_name` (الدافع/المستلم), `received_by` (id الموظف), `note`
   - `is_void` (0/1) + `void_reason` + `void_user` + `void_date` — للإلغاء (لا حذف)
   - `create_user`, `created_at`
2. **جدول الصندوق `cashbox_ledger`** (قيد لكل حركة — append-only، على نمط `ai_audit_log`):
   - `entry_id`, `receipt_id` FK, `source_type`, `source_id`, `direction`, `amount`,
     `balance_after` (الرصيد التراكمي)، `change_user`, `change_at`, `note`.
   - كل قبض = قيد `in`. كل إلغاء = قيد `out` عكسي جديد (لا تعديل/حذف القيد الأصلي).
3. **تعديل مسار `toggle_rentpay()` (وما يماثله لبقية الأنواع):**
   - `غير مدفوع → مدفوع`: يفتح نافذة إنشاء سند قبض (المبلغ الافتراضي = المجدول، التاريخ = اليوم،
     المستلم = المستخدم الحالي، الدافع) → عند الحفظ: إدراج سند + قيد `in` في الصندوق + تحديث الحالة + `paid_date`.
   - `مدفوع → غير مدفوع`: **يتطلب سبب إلزامي** → تعليم السند `is_void=1` + قيد `out` عكسي + تحديث الحالة.
4. **صفحة الصندوق `/cashbox`:** كشف حركة تراكمي (كل قبض/إلغاء، المبلغ، الرصيد، الموظف، الوقت، رابط السند)
   مع فلاتر بالتاريخ والنوع والموظف.
5. **طباعة السند:** `GET /cashbox/receipt/{id}/print` → PDF (DOMPDF) برقم متسلسل وبيانات الشركة (شركة صباح النور).

### القالب المرجعي (موجود بالكود)
- `ai_audit_log` (`database/migrations/2026_06_23_000026_create_ai_audit_log_table.php`) — نموذج الجدول append-only.
- نمط `*_history` (مثل `shop_note_history` — `ShopController.php:1450`) — عرف السجل الدائم بالمشروع.
- واجهات `expense`/`accountings`/`financial` controllers — نموذج واجهة مستند مالي.

### ملاحظة تصميم
`shop_rentpay.rentpay_price` هو المبلغ **المجدول** فقط؛ لا يوجد حقل للمبلغ **المقبوض فعلاً**.
السند يلتقط المقبوض فعلاً بشكل مستقل (يسمح بالدفعات الجزئية مستقبلاً).

## المهمة 2 — صلاحيات إظهار/إخفاء ميزات الذكاء الاصطناعي

### القرارات المعتمدة
- **الميزات المغطّاة:** استخراج عقود الإيجار + قراءة فواتير المشتريات + إعدادات الـ AI + (صلاحية جامعة لكل الـ AI).
- **الآلية:** **صلاحية لكل ميزة** عبر نظام الصلاحيات الحالي (`per_function` / `Perm::get_function_access()`).

### المتطلبات
1. إضافة صفوف `per_function` جديدة (أرقام function جديدة) لكل ميزة AI:
   - AI: استخراج عقود الإيجار (زر/تدفق `ShopAiExtractor` / `LeaseController`).
   - AI: قراءة فواتير المشتريات (`PurchaseController` AI).
   - AI: إعدادات الذكاء الاصطناعي (مفتاح Gemini + الموديل في الإعدادات).
   - AI: صلاحية جامعة (تتحكم بكل ما سبق دفعة واحدة).
2. حراسة كل نقطة دخول AI (زر/route/قسم في الواجهة) بـ `Perm::get_function_access(<الرقم>)` —
   الموظف بدون الصلاحية لا يرى الزر ولا ينفّذ الـ route (تُرفض من الطرفين: واجهة + سيرفر).
3. المدير (`emp_job == 1`) يوزّع هذه الصلاحيات على الأدوار من شاشة الصلاحيات الحالية.

### ملاحظة تصميم
تُطبّق الحراسة على **الواجهة والسيرفر معاً** — إخفاء الزر وحده غير كافٍ؛ لازم الـ route يرفض الطلب.

## المهمة 3 — سجل نشاط شامل للموظفين

### القرارات المعتمدة
- **النطاق:** **كل النظام** — سجل شامل لما يفعله كل موظف.

### المتطلبات
1. **جدول `activity_log`** (append-only): `id`, `user_id`, `action` (create|update|delete|view|login|...),
   `entity_type`, `entity_id`, `summary`, `ip`, `created_at`.
2. التقاط الأحداث تلقائياً على مستوى النظام (middleware + Eloquent/DB events حيثما أمكن) بدل ترصيع كل controller يدوياً.
3. شاشة `/activity-log` للمدير: فلترة بالموظف/التاريخ/نوع الحدث/الجدول، بحث، وترقيم صفحات.

### قرار تصميمي مطلوب حسمه قبل التنفيذ (⚠️)
«كل النظام شامل» — هل نسجّل **القراءات/فتح الصفحات (view)** أيضاً، أم **الكتابة فقط** (إنشاء/تعديل/حذف)؟
- تسجيل كل «view» ينفخ الجدول بسرعة هائلة (كل تصفّح صفحة = صف) ويثقل قاعدة البيانات (Oracle/MySQL على السيرفر).
- **التوصية:** الكتابة (create/update/delete) + تسجيل الدخول/الخروج، مع استثناء القراءات — يغطي «مين غيّر إيش» بدون تضخّم.
- إن أصرّ العميل على تسجيل المشاهدات، نضيفها لاحقاً كخيار مع سياسة تقليم (retention) دورية.

## اعتبارات عامة (كل المهام)
- **قاعدة البيانات:** الجداول القديمة (`shop_rentpay`) بدون migration إنشاء — نستخدم migrations إضافية محروسة بـ `hasColumn`/`hasTable` (نفس نمط `2026_07_18_120000_add_paid_status_to_shop_rentpay.php`).
- **التوافق مع Oracle + MySQL:** التطبيق يدعم الاثنين — نتجنّب صيغ SQL خاصة بمحرّك واحد.
- **الترحيل للسيرفر:** نشر يدوي عبر `git pull` على MyContabo ثم `php artisan migrate` بمستخدم `www-data`،
  إعادة تشغيل `abd-queue` عند اللزوم، وضبط ملكية الملفات (chown). السيرفر كان متأخّراً سابقاً — نتحقق من التزامن أولاً.
- **الأمان:** كل الجداول الجديدة append-only؛ لا حذف مادّي لسندات القبض ولا لقيود الصندوق ولا لسجل النشاط.
